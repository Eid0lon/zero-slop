# Composition Playbook

## Senior Design Sequence

1. Establish the object model.
   - What objects exist? Project, issue, customer, invoice, product, file, route, deployment, report?
   - Layout should reveal relationships between those objects.

2. Choose the first viewport.
   - For apps: usable workflow, not a marketing hero.
   - For marketing: product/object signal plus next-section preview.
   - For commerce: product evidence and comparison path.

3. Build the hierarchy.
   - Use position, size, weight, contrast, grouping, spacing, and copy.
   - Make primary action visually dominant, secondary action quiet, destructive action cautious.

4. Create tokens before decoration.
   - Use semantic surfaces, text, border, focus, accent, status, chart, radius, shadow, and motion tokens.
   - Use raw values only for one-off art direction, and promote repeated values into tokens.

5. Fit the domain.
   - Operational: compact, clear, calm.
   - Developer: precise, artifact-driven, keyboard-friendly.
   - Editorial: typographic, image-led, paced.
   - Commerce: inspectable, trustworthy, comparison-friendly.

## Typography

- Choose type for reading behavior, not trend.
- Keep body readable; do not use viewport-scaled font sizes.
- Use line length and line height to control pace.
- Avoid negative letter spacing.
- Reserve huge type for true heroes, not panels/cards.
- Use mono for technical data, IDs, code, measurements, and status when it adds precision.

## Layout

- Use stable dimensions for boards, grids, tiles, counters, media, and toolbars.
- Avoid nested cards and decorative page-section cards.
- Align controls with the data they affect.
- Make next actions visible near the moment of decision.
- Let repeated components vary only when the content meaning varies.

## Palette

- Start from background, text, border, and one accent.
- Add status and chart colors only when data needs them.
- Avoid one-hue domination unless brand requires it.
- Use accent sparingly enough that it means something.
- Ensure focus and status colors work on every surface.

## Shape and Elevation

- Radius should communicate system personality and hierarchy.
- Controls usually sit tighter than cards/panels.
- Shadows should describe layering, not decorate.
- Borders often do more premium work than heavy shadows.
- Child radius should not exceed parent radius.

## Assets and Media

- Prefer real product screenshots, real data, real places, real objects, or generated images that directly show the subject.
- Avoid abstract filler where inspection matters.
- Give media explicit dimensions or aspect ratios.
- Keep image style coherent across the page.

## Copy

- Button labels describe the next action.
- Headings name the object, outcome, or decision.
- Empty states name why nothing is shown and how to recover.
- Errors name the problem and next step.
- Claims need evidence, period, unit, or source.
