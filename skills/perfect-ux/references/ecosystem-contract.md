# Ecosystem Contract: perfect-ux

## Role

Perfect UX owns user success: jobs, journeys, information architecture, interaction clarity, forms, feedback, recovery, accessibility-as-operability, content clarity, trust, performance feel, and measurement.

It does not replace the visual craft owned by `perfect-design`, the anti-generic detection owned by `no-slop`, or the real implementation boundary owned by `reality-skill`.

## Composes With

- `perfect-design`: when UX improvements require layout, hierarchy, visual direction, tokens, art direction, or premium UI craft.
- `no-slop`: before and after material UI work to block generic AI residue and weak visual/copy defaults.
- `reality-skill`: when UX depends on real actions, data, persistence, integrations, validation, local storage, backend behavior, or honest mocks.
- `review`: when UX changes touch auth, billing, privacy, security, compliance, data contracts, or irreversible actions.
- `debug`: when friction comes from runtime state, hydration, async failures, routing, focus bugs, browser behavior, or performance defects.
- `create-deep-search`: when current domain-specific UX research or regulation is required.

## Ownership Boundary

Perfect UX can:

- map journeys and jobs
- audit UX gates
- patch copy, IA, controls, forms, states, recovery paths, focus, keyboard behavior, and responsive UX
- define measurement plans and usability test scripts
- judge UX quality with subagents
- block weak results

Perfect UX should not:

- redesign visual identity broadly without `perfect-design`
- claim real workflow completion without `reality-skill` when data/actions are fake
- invent research findings, user quotes, analytics, compliance, or proof
- add heavy dependencies when existing primitives are sufficient
- optimize conversion by obscuring risk, consent, cost, cancellation, or user control

## Quality Gate

Done means:

- UX Contract exists or was inferred.
- Primary journey is mapped.
- UX gates pass or blockers are named.
- Accessibility, keyboard, responsive, error, feedback, trust, and recovery checks completed or limitations stated.
- Sibling skills used when their ownership applies.
- Verification evidence recorded.
- Remaining risk named as user harm, not internal inconvenience.
